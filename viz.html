Parent-Child 检索模式

为什么用 Parent-Child？

  • Child 小块 → 精准向量匹配
  • Parent 大块 → 完整上下文给 LLM
  • 图片 IDs 绑定在 Parent 上
  • 避免截断关键信息

数据结构

Child: { child_id: "空调_p1_c0", parent_id: "空调_p1", text: "...(≤1000字)", image_ids: ["AC_001", ...] } Parent: { parent_id: "空调_p1", text: "...(≤1500字)", image_ids: ["AC_001", ...] }

Validator 检查项

  • used_images ≤ MAX_IMAGES (5)
  • 数值单位可溯源到检索文本
  • Claude 覆盖度检查

AnswerGenerator

  • 统计图片频率，取 Top-5
  • 构建带编号的上下文
  • 发送 Base64 图片给 Claude
  • 返回 JSON: answer + image_ids

产品识别三级回退

  • L1 QuestionAnalyzer 直接识别产品名
  • L2 term_product_map 字符串匹配
  • L3 Claude LLM 推断产品名
match_product_by_term(query) → O(n) scan term_product_map.json lookup_product_by_term(query) → Claude call with full term map → returns product_name | null

依赖子问题处理

  • sub_q_dependent=True → 顺序执行
  • 每次查询附加已积累的章节标题
  • sub_q_dependent=False → 简单循环
  • 按 parent_id 去重

Policy 检索

  • 仅 Dense 检索（无 Sparse）
  • 集合: policy
  • 来源: policy/*.md 文件
  • top_k=3，返回原始文本
  • 处理退换货、物流等通用问题
